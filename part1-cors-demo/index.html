üìö Context: The Three Barriers

In Part 0, we set up the gateway to handle CORS, macaroons, and TLS. But why are these issues? Let's understand what you're about to experience:

üõ°Ô∏è CORS (Cross-Origin Resource Sharing)

What it is: A browser security feature that blocks web pages from making requests to a different domain/port than the one that served the page.

Why it exists: Prevents malicious websites from accessing your sensitive APIs without permission.

The problem: Your web app (http://localhost:8999) trying to call TAPD (https://127.0.0.1:8289) = different origins = blocked!

üç™ Macaroon Authentication

What it is: A cryptographic credential (like a fancy cookie) that proves you're authorized to use TAPD/LND APIs.

Why it exists: Security! Only authorized applications can create assets, send transactions, etc.

The problem: Macaroons are binary files stored on your file system. Web browsers can't read local files for security reasons, and you definitely shouldn't expose admin credentials to frontend code!

üîí TLS Certificates

What it is: HTTPS encryption that secures communication between your app and the server.

Why Polar uses self-signed certs: Development environments generate their own certificates instead of using a Certificate Authority (like Let's Encrypt).

The problem: Browsers reject self-signed certificates by default because they can't verify their authenticity. This is the "Your connection is not private" warning you see.

üí° The Gateway Solution: The REST gateway from Part 0 sits between your web app and TAPD. It runs as a server (not in the browser), so it can read macaroon files, handle TLS certificates, and add proper CORS headers. This is why we need it!
